01 Qu'est-ce que le Corewar ?

Le Corewar est un jeu dans lequel des programmes — appelés champions — s'affrontent à l'intérieur d'une machine virtuelle. Chaque champion est un bytecode qui s'exécute dans un espace mémoire partagé : l'arène. Le but est d'être le dernier processus en vie.

Le projet se décompose en trois parties que vous devez rendre :

⚙️
La Machine Virtuelle
L'arène dans laquelle les champions s'exécutent. Elle gère la mémoire, les processus, le cycle de vie et détermine le vainqueur.
🔧
L'Assembleur
Compile les fichiers .s (assembleur lisible) en bytecode .cor exécutable par la VM. Parsing, résolution de labels, encodage.
⚔️
Le Champion
Un programme en assembleur Corewar capable de survivre. Il doit faire des live avec son numéro pour rester en vie.

Le flux de données est simple : vous écrivez un champion en .s, l'assembleur le compile en .cor, et la VM charge et exécute les champions :

champion.s
→
Assembleur (asm)
→
champion.cor
→
Machine Virtuelle (corewar)
→
Arene 4096 octets

03 L'Arene — Mémoire circulaire

L'arène est un tableau de 4096 octets (MEM_SIZE). Chaque octet peut contenir une instruction ou une donnée. La mémoire est circulaire : l'adresse 4096 est la même que l'adresse 0. Tous les adressages sont modulo MEM_SIZE.

En plus du bytecode, la VM maintient deux tableaux parallèles : owner[] qui indique quel joueur possède chaque octet (pour l'affichage couleur), et scrb[] qui marque les écritures récentes (surbrillance ncurses).

Big Endian — Pourquoi c'est important

La VM Corewar est Big Endian : l'octet de poids fort est stocké en premier. Quand vous lisez 4 octets depuis la mémoire pour former un int, le premier octet est le plus significatif. C'est l'inverse de l'architecture x86 (Little Endian).

Valeur 0x0EA83F3 stockée en mémoire :

0E
+0
A8
+1
3F
+2
03
+3

Big Endian : l'octet le plus significatif (0E) est à l'adresse la plus basse

En C, on utilise reverse_bytes() pour lire correctement les valeurs Big Endian depuis le tableau d'octets. En Rust, on utilise i32::from_be_bytes(), ce qui est plus sûr et idiomatique :

Lecture Big Endian 
// C — type punning (marche sur x86, UB en strict C)
int reverse_bytes(t_vm *vm, int pc, int size) {
    t_uchar bytes[4] = {0};
    for (int i = 0; i < size; i++)
        bytes[i] = vm->ram.arena[(pc + i) % MEM_SIZE];
    return *(int*)&bytes[0]; // Type punning !
}
// Rust — parsing sur, pas de type punning
fn read_mem(&self, pc: usize, size: usize) -> i32 {
    let mut bytes = [0u8; 4];
    for i in 0..size {
        bytes[4 - size + i] = self.arena[(pc + i) % MEM_SIZE];
    }
    i32::from_be_bytes(bytes) // Safe et idiomatique
}

Fork — Créer de nouveaux processus

Les instructions fork et lfork créent un nouveau processus qui hérite de tous les registres et du carry du parent. Seul le PC diffère : il est positionné à PC + (param % IDX_MOD) pour fork, ou PC + param (sans modulo) pour lfork.

fork vs lfork 
// fork : avec IDX_MOD
void op_fork(t_vm *vm, t_list *process) {
    int move = reverse_bytes(vm, P->pc + 1, 2);
    add_process(vm, process, mem_mod(P->pc + (move % IDX_MOD)));
    P->pc = mem_mod(P->pc + 3);
}

// lfork : SANS IDX_MOD (long fork)
void op_lfork(t_vm *vm, t_list *process) {
    int move = reverse_bytes(vm, P->pc + 1, 2);
    add_process(vm, process, mem_mod(P->pc + move)); // Pas de % IDX_MOD !
    P->pc = mem_mod(P->pc + 3);
}
// Rust — corewar-vm/src/vm.rs
fn op_fork(&mut self, idx: usize) {
    let move = self.read_mem(pc + 1, 2);
    let new_pc = (pc as i32 + move % IDX_MOD).rem_euclid(MEM_SIZE as i32);
    self.new_processes.push(clone_process(idx, new_pc as usize));
    self.processes[idx].pc = (pc + 3) % MEM_SIZE;
}

fn op_lfork(&mut self, idx: usize) {
    let move = self.read_mem(pc + 1, 2);
    let new_pc = (pc as i32 + move).rem_euclid(MEM_SIZE as i32); // Pas de IDX_MOD !
    self.new_processes.push(clone_process(idx, new_pc as usize));
    self.processes[idx].pc = (pc + 3) % MEM_SIZE;
}

05 Cycle de vie d'une instruction

Chaque instruction passe par 4 phases. La notion clé est que les instructions consomment toujours leurs cycles : même si l'instruction est invalide, les cycles sont déduits. Une instruction ne s'exécute qu'à la fin de son dernier cycle.

1
Fetch — Lecture de l'opcode
Le processus lit l'octet à son PC dans l'arène. Si c'est un opcode valide (1-16), il stocke l'index dans ir et initialise duration avec le nombre de cycles de l'instruction. Sinon, il avance le PC de 1.
2
Decode — Attente des cycles
À chaque cycle, le compteur duration est décrémenté. L'instruction attend sa durée complète (ex: live = 10 cycles, fork = 800 cycles). Pendant ce temps, les autres processus continuent.
3
Validate — Vérification des paramètres
Quand duration atteint 0, l'instruction est prête. La VM vérifie l'OCP (octet de codage des paramètres) et valide que les types de paramètres correspondent à ceux attendus par l'opcode. Si invalide, l'instruction est ignorée mais le PC avance.
4
Execute — Exécution et mise à jour
L'instruction est exécutée : modification des registres, écriture mémoire, saut du PC, création de processus (fork), etc. Le PC avance au-delà de l'instruction complète, et ir est remis à -1 pour la prochaine instruction.

La boucle principale

Boucle principale 
// C — vm/src/computer-system/run_cpu.c
int run_processes(t_vm *vm, int control) {
    while (vm->processes != NULL && vm->cycle_to_die > 0) {
        // Verification CYCLE_TO_DIE ?
        int check = (vm->cycles && (vm->cycles - vm->diff_to_die)
                                         % vm->cycle_to_die == 0);
        update_cycles(vm);             // Incremente, decremente CYCLE_DELTA
        process_operations(vm, vm->processes);  // Execute tous les processus
        kill_zombies(vm, vm->processes, check); // Tue les processus sans live
    }
    // Annoncer le gagnant...
}
// Rust — corewar-vm/src/vm.rs
pub fn run(&mut self) {
    while !self.processes.is_empty() && self.cycle_to_die > 0 {
        let check = self.cycles > 0
            && (self.cycles - self.diff_to_die) % self.cycle_to_die == 0;
        self.update_cycles();
        self.process_operations();
        self.kill_zombies(check);
    }
}

CYCLE_TO_DIE — Le mécanisme de survie

Tous les CYCLE_TO_DIE cycles, la VM vérifie que chaque processus a fait au moins un live. Ceux qui n'en ont pas fait sont tués (zombies). Ce mécanisme est ce qui fait que la partie finit par se terminer :

📖 Détail du mécanisme CYCLE_TO_DIE ▶

1. Tous les CYCLE_TO_DIE cycles, on vérifie les lives.

2. Si le nombre total de live (variable nlives) est ≥ NBR_LIVE (21), alors CYCLE_TO_DIE est décrémenté de CYCLE_DELTA (50). La partie s'accélère.

3. Si après MAX_CHECKS (10) vérifications consécutives sans assez de lives, on décrémente quand même CYCLE_TO_DIE.

4. Quand CYCLE_TO_DIE atteint 0 ou moins, tous les processus sont tués et la partie se termine.

5. Le gagnant est le dernier joueur à avoir été rapporté comme "en vie" par l'instruction live.

Instructions clés en détail

live — Signaler qu'un joueur est en vie ▶

Prend un paramètre direct de 4 octets. Incrémente live_count du processus et nlives global. Si le numéro correspond à un joueur, celui-ci est déclaré "en vie". Le PC avance de 5 (1 opcode + 4 octets). Pas d'OCP.

live
// C
void op_live(t_vm *vm, t_list *process) {
    P->live_count++;
    vm->nlives++;
    int live = reverse_bytes(vm, P->pc + 1, 4);
    player_alive(vm, live);
    P->pc = mem_mod(P->pc + 5);
}
zjmp — Saut conditionnel si carry = 1 ▶

Prend un index direct de 2 octets. Pas d'OCP. Si le carry est à 1, le PC saute à PC + (param % IDX_MOD). Sinon, le PC avance de 3 (1 opcode + 2 octets). C'est l'instruction de contrôle de flux principale, utilisée pour les boucles et les conditions.

zjmp
// C
void op_zjmp(t_vm *vm, t_list *process) {
    int move = reverse_bytes(vm, P->pc + 1, 2) % IDX_MOD;
    P->pc = mem_mod(P->pc + (P->carry ? move : 3));
}
sti — Stockage indexé ▶

Prend un registre et deux index. Additionne les deux derniers, utilise cette somme comme adresse, et y copie la valeur du registre. Les paramètres indirects sont modulo IDX_MOD. L'adresse de stockage est relative au PC, et la VM écrit 4 octets via store_at. Piège : store_at écrit les octets de l'adresse la plus haute vers la plus basse, il faut donc ajouter +3 à l'adresse calculée.

ld / lld — Chargement avec/sans IDX_MOD ▶

ld charge une valeur dans un registre. Si le paramètre est indirect, l'adresse est modulo IDX_MOD. lld (long load) fait la même chose sans IDX_MOD, ce qui permet d'accéder à n'importe quelle adresse dans l'arène. Les deux modifient le carry : carry = 1 si le résultat est 0, sinon carry = 0.

L'OCP — Octet de Codage des Paramètres

L'OCP (ou ACB) encode le type de chaque paramètre sur 2 bits : 01 = registre, 10 = direct, 11 = indirect. Les bits inutilisés sont à 0.

Exemple : sti r1, %:live, %1

r1 → 01
%:live → 10
%1 → 10
→
01 10 10 00 = 0x68

Calcul de l'OCP en code

OCP (ACB) 
// C — assembler/src/synthesis/collect_codebytes.c
static char get_acb(t_instruction *instruction) {
    t_args *argument = instruction->args;
    int i = 7, acb = 0;
    while (argument) {
        if (argument->reg && i--)
            acb += ft_power(2, i);                       // 01 = registre
        else if (argument->dir && i--)
            acb += ft_power(2, i + 1);                 // 10 = direct
        else if (argument->ind && i--)
            acb += ft_power(2, i + 1) + ft_power(2, i); // 11 = indirect
        i--;
        argument = argument->next;
    }
    return (acb);
}
// Rust — corewar-asm/src/codegen.rs
fn compute_acb(instr: &Instruction) -> u8 {
    let mut acb: u8 = 0;
    let mut bit_pos: i32 = 7;
    for arg in &instr.args {
        bit_pos -= 1;
        match arg {
            Arg::Reg(_) => { acb |= 1 << bit_pos; }     // 01
            Arg::Dir(_) => { acb |= 2 << bit_pos; }     // 10
            Arg::Ind(_) => { acb |= 3 << bit_pos; }     // 11
        }
        bit_pos -= 1;
    }
    acb
}

08 L'Assembleur — Pipeline complet

L'assembleur transforme un fichier .s en .cor en plusieurs étapes. Le principe est le même en C et en Rust, mais l'implémentation diffère dans la gestion des erreurs et des structures de données.

1
Lexer — Tokenisation
Découpe le fichier en tokens : labels, opcodes, paramètres, commentaires. Vérifie la syntaxe de base (caractères autorisés dans les labels, séparateurs valides).
2
Parser — Validation sémantique
Valide que chaque instruction a le bon nombre de paramètres et les bons types. Vérifie que tous les labels référencés existent. Construit la structure interne (liste chaînée en C, Vec<Label> en Rust).
3
Calcul des positions — Première passe
Parcourt toutes les instructions pour calculer leur position en mémoire (mem_pos) et la taille totale du programme. Chaque instruction a une taille variable qui dépend de l'OCP et des types de paramètres.
4
Codegen — Deuxième passe
Résout les labels (calcule les offsets relatifs) et génère le bytecode final. Écrit le header + le bytecode dans le fichier .cor. Toutes les valeurs sont en Big Endian.

Résolution des labels

Un label référencé par %:label est résolu en calculant la différence de position entre l'instruction qui référence et l'instruction cible :

Label resolution 
// C — assembler/src/analysis/parsing.c
int exist_label(t_code *code, char *label, int line) {
    while (code) {
        if (ft_strcmp(label, code->label) == 0)
            return (1); // Label trouvé
        code = code->next;
    }
    // Erreur : label inconnu
    return (0);
}
// Rust — corewar-asm/src/parser.rs
pub fn validate_labels(code_tab: &[Label]) -> Result<(), AsmError> {
    let defined: Vec<&str> = code_tab.iter().map(|l| l.label.as_str()).collect();
    for label in code_tab {
        for instr in &label.instructions {
            for arg in &instr.args {
                match arg {
                    Arg::Dir(s) if s.starts_with(':') => {
                        if !defined.contains(&&s[1..]) {
                            return Err(AsmError::Label(...));
                        }
                    }
                    // idem pour Arg::Ind
                    _ => {}
                }
            }
        }
    }
    Ok(())
}

09 C vs Rust — Comparaison d'implémentation

Le même projet implémenté en C et en Rust illustre les différences fondamentales entre les deux langages. Voici les choix architecturaux et leurs conséquences :

AspectCRustImpact
Processus t_list* (liste chaînée, malloc par nœud) Vec<Process> (tableau contigu) Rust 5-21× plus rapide (localité cache)
Dispatch Pointeurs de fonction dans g_op_tab[].f match ir { 0 => ..., 15 => ... } Équivalent (les deux sont O(1))
Lecture mémoire *(int*)&bytes[0] (type punning, UB) i32::from_be_bytes() (safe) Rust (pas d'UB)
Gestion erreurs Codes de retour -1, ft_printf_fd(2,...) Result<T, E> + ? + thiserror Rust (impossible d'oublier)
Fork insertion Insère immédiatement dans la liste Buffer new_processes, insère après le cycle C (comportement natif du sujet)
Débordement entier UB en C standard wrapping_add() / wrapping_sub() Rust (comportement défini)
Taille binaire 231 Ko 1.2 Mo C (5× plus petit)
Architecture Monolithique + libft 4 crates (common, asm, vm, disasm) Rust (modulaire)

Pourquoi le Rust est 5-21× plus rapide ?

🧱
Vec<Process> vs liste chaînée
Le Rust stocke les processus dans un tableau contigu en mémoire. Le CPU peut précharger les données (cache lines). Le C utilise des nœuds éparpillés avec malloc individuel = cache misses constants.
♻️
Pas de malloc/free par fork
Le C alloue et libère un nœud de liste chaînée pour chaque fork et chaque zombie tué. Le Rust utilise un Vec qui recycle la mémoire via retain_mut.
⚡
Optimisations LLVM
Le compilateur Rust (backend LLVM) est plus agressif : inlining automatique, vectorisation SIMD, élimination de code mort. Même avec -O2, GCC ne matche pas LLVM sur cette base de code.

Kill zombies — La différence algorithmique

kill_zombies 
// C — parcours de liste + suppression de nœud
void kill_zombies(t_vm *vm, t_list *process, int check) {
    while (process) {
        if (!P->live_count || vm->cycle_to_die < 0) {
            // Supprimer le nœud de la liste chaînée
            // + free() le nœud et le processus
            // Complexité : O(N) par suppression × N processus
            vm->process_alive--;
        }
        P->live_count = 0;
        process = process->next;
    }
}
// Rust — retain_mut en O(N) total
fn kill_zombies(&mut self, check: bool) {
    if self.cycles > 0 && check {
        let cycle_to_die = self.cycle_to_die;
        self.processes.retain_mut(|p| {
            if p.live_count == 0 || cycle_to_die < 0 { false }
            else { p.live_count = 0; true }
        });
        self.process_alive = self.processes.len() as i32;
    }
}
